Grails

É um framework Java

mas é limpinho :)

Uilian Souza / @uilian

Sem pessimismo, porque se tiver que usar um framework, eu sugiro que use o Grails. Ele praticamente fica fora do seu caminho o tempo todo, e está lá (quase sempre) para ajudar quando necessário. Como todo bom prestador de serviços deveria ser.

Porque vale a pena aprender

  • Framework Web MVC similar a RoR, CodeIgniter, Django e Play!
  • Full-Stack
  • Convention-Over-Configuration
  • O ciclo de desenvolvimento é rápido
  • Curva de aprendizado baixa
  • Extensível via plugins, e existem muitos disponíveis
  • Aplicar alterações em configurações e fontes não obriga a reiniciar o servidor
  • Bem documentado (online e livros)
Estes são os pontos que melhor definem e diferenciam o Grails para mim, vamos voltar neste pontos adiante. Mas antes, precisamos falar um pouco sobre o Groovy.

Ou seja, o difícil não é fácil

  • Atualizar a versão do framework pode ser uma dor de cabeça.
  • O framework decide muita coisa por você.
  • IDEs às vezes mais atrapalham que ajudam.
  • Dilema do Peter Parker.
  • Grails ainda é focado apenas em servlet containers.(suporte ao Netty está previsto na v3.0)
  • É obrigatório extremamente indicado aprender Groovy, GORM, Gradle e GVM.
1- É especialmente verdadeiro quando se utilizam muitos plugins. As atualizações do framework não mantém compatibilidade com versões anteriores.
2- O que é bom e ruim ao mesmo tempo. Se bem utilizado, Grails pode dar um ganho de produtividade considerável. Mas confiança cega, super-utilização, e a falta de conhecimento vão gerando débito técnico pelo caminho, como qualquer outro framework.
3- As IDEs que oferecem bom suporte à Groovy e Grails são Intellij IDEA e GGTS. Mas eu aconselho utilizar apenas quando realmente for necessário, principalmente enquanto estiver aprendendo a linguagem. Principalmente o GGTS, que é baseado em Eclipse, gera muito lixo por baixo dos panos, que é necessário para seu funcionamento, e nada tem a ver com Grails. IDEA só tem suporte ao Grails na versão paga.
4- Com grandes poderes vem grandes responsabilidades. O uso indiscriminado de "def" outras estruturas "groovyanas" podem gerar código obscuro e díficil de manter.
5- Na prática, isso significa que para programadores que tenham escalabilidade como pré-requisito talvez devessem considerar outras opções antes do Grails.
6- Existe um ecosistema todo relacionado ao Grails, e para tirar vantagem real desta plataforma, é melhor investir tempo para aprender todas as ferramentas.

Será uma breve introdução às features mais interessantes do Groovy, com foco naquelas que aparecem mais frequentemente no uso do Grails.

É uma linguagem:

  • Inspirada em linguagens como Python, Ruby e Smalltalk
  • Que roda sobre a JVM
  • Extende as funcionalidades do Java
  • Dinâmica (type-checking E metaprogramming)
  • Permite checagem de tipos estática
Todas estas características tem seu lado bom e seu lado ruim. Como tudo na vida. Groovy tem evoluído para tentar equilibrar flexibilidade, produtividade e performance.

Integração de "mão dupla" com Java

1 - Permite o uso de bibliotecas e código Java

2 - Código fonte Groovy compila para bytecode Java

O ponto aqui é ressaltar que muita coisa já existe pronta em Java, e pode ser utilizada sem modificações em Groovy. Além disso, artefatos construídos com Groovy também podem ser utilizados em outros programas Java. Ex: Pentaho (PDI, PRD).

Closures


def quadrado = {it**2}     // parâmetro implícito
println quadrado(5)        // retorna 25

def soma = { a, b -> a+b }
println soma(4, 10)        // retorna 14
  • Bloco anônimo de código delimitado por chaves.
  • Tem acesso ao estado do escopo onde ele é executado.
  • Maximiza o encapsulamento e expressividade / legibilidade.

Groovy Truth

Regra: tudo que não for nulo, vazio, zero (ou explicitamente FALSE, óbvio), será avaliado para TRUE 


// True                      | // False
def a = true                 | def b = false
def numbers_full = [1,2,3]   | def numbers = []
def s = 'abc'                | def s = ''
def n = 1                    | def n = 0
def obj = new Object()       | def obj = null

Traits e Mixins


trait FlyingAbility { String fly() { "I'm flying!" } }
trait WalkAbility { String walk() { "I'm walking!" } }

class Thing implements FlyingAbility { }
def a = new Thing()
assert a.fly() == "I'm flying!"

class Thing { }
def b = new Thing() as FlyingAbility
assert b.fly() == "I'm flying!"

def c = new Thing()
def d = c.withTraits FlyingAbility, WalkAbility
d.fly(); d.walk()

@Mixins estão deprecated a partir da versão 2.3, em favor dos Traits.

O básico para ser compreendido aqui é que é possível extender as funcionalidades de objetos Groovy através de composição em tempo de execução. Cool!

Criando uma aplicação Grails


// Cria a aplicação
~ grails create-app MinhaApp
                        

// Executa a aplicação
 ~ cd MinhaApp
 ~ grails run-app MinhaApp
                        

//// Convention-over-configuration
|- grails-app
|--- assets          // Conteúdo gerenciado pelo Asset Plugin
|--- conf            // Arquivos de configuração
|--- controllers
|--- domain
|--- i18n            // Internacionalização
|--- services
|--- taglib
|--- utils
|--- views           // Telas, templates
|- src
|--- groovy
|--- java
|- test              // Testes unitários
|- web-app           // Imagens, CSS, JS, HTML ...
O que ajustar:
- O arquivo conf/Config.groovy, opções de log
- Falar sobre o Asset Plugin, e que ele substitui o Resources Plugin.

Views

  • GSP (HTML mesclado com taglibs próprias e/ou do framework)
  • Conceito de templates
  • Permite scripts em Groovy
  • Taglibs são muito simples
  • Scaffolding
Templates ficam na pasta views/layout/, podem ser personalizados de acordo com as necessidades do sistema, e até mesmo criar vários.
Scaffolding também pode ser personalizado, como vamos ver mais adiante.
O código das views é relativamente limpo, e é uma boa prática (IMO) utilizar o mínimo possível das "facilidades" do framework nesta parte de interface, principalmente plugins. Deixando claro que os frameworks em geral não são "O PROBLEMA". O problema são os programadores que os utilizam como muletas para compensar suas próprias deficiências. Ex: Não use uma taglib de ajax porque não sabe javascript, APRENDA javascript. Não use ORM porque não sabe SQL, use porque entende as vantagens.

~ grails generate-views [domain-class-name]
~ grails generate-all [domain-class-name]

// Customizar os templates:
~ grails install-templates
O scaffolding era uma operação padrão do framework até a versão 2.3, quando passou a ser disponibilizada como um plugin. Existem outros plugins para customizar o comportamento e a aparência da interface, como o Twitter Bootstrap, Fields (para os campos).

Assets pipeline *

  • Otimização e organização do uso de recursos estáticos nas páginas.
  • Atualização on-the-fly, alterações são disponibilizadas sem reload.
  • Redução do tamanho das dependências, através da compressão, minificação e cache (UglifyJS).
  • Dependências e ordem de carga são configuráveis diretamente nos CSS e JS.
  • Suporte à compilação de recursos estáticos LESS, SASS/SCSS, Handlebars, Ember, CoffeScript.

* na versão 2.4 substituiu o plugin Resources

Escopos

  • servletContext -> compartilha entre toda a aplicação
  • session -> apenas na sessão do usuário
  • request -> objetos referentes à requisição corrente
  • params -> mapa contendo os parâmetros passados no request
  • flash -> temporário, mantém o conteúdo apenas até o próximo request

class SomeController {
    def find() {
        def findBy = params["findBy"]
        def appContext = request["foo"]
        def loggedUser = session["logged_user"]
    }
}
Nada demais aqui, parecido com os demais frameworks, legal mostrar a sintaxe, que é mais limpa e direta no Grails.

UrlMappings

  • Por convenção os mapeamentos são definidos no formato /controller/action/id
  • As configurações são armazenadas no arquivo grails-app/conf/UrlMappings.groovy
  • Permite agrupar ações que compartilham caminhos (2.3 em diante)
  • É possível configurar redirecionamentos para URIs
  • Mapeamento REST para controllers por convenção (2.3 em diante)

//
// Mapeamento padrão de uma aplicação recém criada
//
class UrlMappings {
        static mappings = {
        "/$controller/$action?/$id?(.$format)?"{
            constraints {
                // apply constraints here
            }
        }

        "/"(view:"/index")
        "500"(view:'/error')
        }
}

Services

  • Declarados em grails-app/services/
  • Injetados nas classes que os utilizam
  • Transacionais por padrão
  • Singletons

Dê preferência por serviços para orquestrar interações entre recursos da aplicação (domain objects) ou externas (IO).

Devem sempre que possível ser mantidos stateless, mas é possível alterar o escopo. São Spring Beans, e podemos tirar vantagem disso para configuração e controle de transações. Serviços são utilizados principalmente pensando em remover complexidade da camada de visão. 

class SomeService {
    void someMethod(param) {
        // do something
    }
}
/******************************/
class FirstController {
    def someService

    def someAction() {
        try {
            someService.someMethod(params.id)
        } catch(e) {
            log.error e
        }
    }
}

Persistência

  • GORM == Hibernate sem dor
  • grails-app/conf/DataSource.groovy
  • Dynamic finders
  • Criteria queries
  • Database migrations
  • Herança de domínios
Herança de domínios pode ser de dois tipos: table-per-hierarchy (d) / table-per-subclass

class Book {
    String title
    static belongsTo = Author
    static hasMany = [authors : Author]
    static constraints = {name nullable: false}
}

class Author {
    String name
    static hasMany = [books : Book]
}

def a = new Author(name:"Author One")
a.addToBooks(new Book(title:"Book One"))
a.addToBooks(new Book(title:"Book Two"))
a.save()
                        

// Acesso direto por id
def book = Book.get(1)
// Dynamic Finders
def book = Book.findByTitle("Book One")
def books = Book.findByTitleIlike("Book%")
def books = Book.findByAuthor(Author.get(1))
// Where Queries
def query = Book.where{title like "Book%"}
def aQuery = query.where{authors {name == "Author One"}}
Book b = aQuery.find()
// Criteria Queries
def c = Book.createCriteria()
def results = c.list(max:4, sort:"title") {
    and {
        ilike("title", "Book%")
        authors{
            eq("name", "Author One")
        }
    }
    maxResults(10)
    order("holderLastName", "desc")
}
